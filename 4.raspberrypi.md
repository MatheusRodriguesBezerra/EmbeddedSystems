Quero que você implemente o backend do meu projeto de Sistemas Embutidos chamado “Enigma Machine”.

Contexto do projeto:
- O sistema tem 3 partes: Arduino Mega, Raspberry Pi 3 B+ e app Android.
- O Raspberry Pi é o núcleo lógico do sistema.
- O Arduino funciona como interface física: lê teclas, rotores físicos, botões e mostra LEDs/display.
- O Raspberry Pi recebe eventos do Arduino por Serial USB, executa o algoritmo de cifra estilo Enigma e devolve a saída ao Arduino.
- O Raspberry Pi também se comunica com um app Android por HTTP em Wi-Fi local.
- O objetivo do sistema é permitir troca de mensagens cifradas entre a máquina física (Arduino + Raspberry Pi) e o app mobile, em ambos os sentidos.
- A configuração (ordem dos rotores, posições iniciais, modo e estado de turno) deve ficar sincronizada entre Raspberry Pi e app Android.
- As chaves/configuração não devem ser enviadas junto com cada mensagem cifrada; a configuração é sincronizada previamente.
- A comunicação entre app e Raspberry Pi deve seguir um modelo half-duplex: se uma ponta estiver em SENDING, a outra deve estar em RECEIVING.

Requisitos funcionais:
1. Implementar uma Enigma simplificada com:
   - 3 rotores
   - refletor
   - rotação do rotor da direita a cada letra
   - possibilidade de definir ordem dos rotores
   - possibilidade de definir posição inicial dos rotores
   - modo encrypt/decrypt usando a mesma lógica da Enigma
2. Receber comandos do Arduino por Serial:
   - KEY:<letra>
   - ROTOR:<indice>:<delta>
   - MODE:<ENC|DEC>
   - SEND
   - STATUS
3. Responder ao Arduino por Serial:
   - OUT:<letra>
   - POS:<r1>,<r2>,<r3>
   - STATUS:<mensagem>
4. Expor API HTTP local para comunicação com o app Android.
5. Permitir que o app Android:
   - envie payload cifrado pela rota /message/:payload quando estiver em SENDING
   - receba payload cifrado da máquina física quando estiver em RECEIVING
   - altere configuração dos rotores
   - solicite o estado atual
6. Expor as rotas HTTP:
   - GET /ping (verificação de comunicação)
   - GET /message/:payload (transporte de payload cifrado)
7. Controlar estado de turno da comunicação:
   - IDLE
   - SENDING
   - RECEIVING
   - impedir que Raspberry Pi e app estejam ambos em SENDING ao mesmo tempo
   - alternar os papéis após uma mensagem ser processada com sucesso
8. Guardar o estado atual do sistema em memória e em arquivo JSON.
9. Estruturar o código de forma modular e limpa.

Requisitos não funcionais:
- código em Python 3
- arquitetura modular
- fácil de testar sem o hardware real
- logs claros
- tratamento de erro
- tipagem quando possível
- separar lógica de domínio da comunicação
