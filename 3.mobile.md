Quero que você implemente um app mobile em React Native para meu projeto de Sistemas Embutidos chamado “Enigma Machine”.

Contexto do projeto:
- O sistema tem Arduino Mega, Raspberry Pi 3 B+ e app mobile.
- O Arduino é a interface física.
- O Raspberry Pi executa a lógica da Enigma e mantém o estado central.
- O app mobile se conecta ao Raspberry Pi por HTTP em rede local.
- O objetivo do app é trocar mensagens com a máquina física:
  - cifrar localmente mensagens antes de enviar
  - decifrar localmente mensagens cifradas recebidas
  - configurar rotores
  - sincronizar estado

Requisitos do app:
1. Tela principal com:
   - status da conexão
   - modo atual (encrypt/decrypt)
   - estado de turno do app (IDLE/SENDING/RECEIVING)
   - mensagem recebida
   - último payload cifrado
   - campo para enviar mensagem
   - botão para enviar
   - botão para sincronizar
2. Tela de configuração:
   - definir IP (ou hostname) do Raspberry Pi
   - selecionar 3 rotores
   - definir ordem dos rotores
   - definir posições iniciais
   - escolher modo
   - sincronizar estado de turno com o Raspberry Pi
   - enviar ao Raspberry Pi
3. Estado global do app:
   - conexão
   - endereço do Raspberry Pi
   - config atual
   - estado de turno atual
   - histórico de mensagens
4. Comunicação:
   - usar HTTP como primeira versão
   - endpoint GET /ping para verificar conectividade com o Raspberry Pi
   - endpoint GET /message/:payload para transporte de payload cifrado
   - apenas quem estiver no estado SENDING pode enviar pela rota /message/:payload
   - quem estiver no estado RECEIVING deve decifrar localmente o payload recebido
   - após processar uma mensagem com sucesso, app e Raspberry Pi devem alternar os papéis SENDING/RECEIVING
5. Apenas 3 rotores fixos (I, II, III), sem criação de novos rotores no app
6. Código limpo e organizado
7. TypeScript
8. arquitetura simples e profissional

Requisitos de UX:
- interface simples e clara
- foco em demonstração académica
- sem excesso de design
- mostrar claramente:
  - mensagem recebida
  - mensagem enviada
  - payload cifrado transmitido
  - rotores ativos
  - posições atuais
  - estado de turno (IDLE/SENDING/RECEIVING)
  - estado de sincronização

Protocolo esperado com o Raspberry Pi:
- rota GET /ping
  - resposta esperada: { "status": "ok", "connectedArduino": true }
- rota GET /message/:payload
  - o payload deve ser uma mensagem já cifrada localmente pela máquina que está em SENDING
  - exemplo de requisição: /message/NCP
  - resposta esperada: { "status": "received", "payload": "NCP", "messageId": "..." }
- rota de configuração (exemplo sugerido: POST /config)
  - payload: { "order": ["I","II","III"], "positions": [5,3,10], "mode": "ENC", "role": "SENDING" }
  - resposta: estado atualizado da máquina
- regra de sincronização:
  - app e Raspberry Pi devem iniciar com a mesma configuração e posições
  - quem envia cifra localmente e avança os rotores
  - quem recebe decifra localmente o payload e avança os rotores da mesma forma
  - o sistema deve evitar que app e Raspberry Pi estejam ambos em SENDING ao mesmo tempo
